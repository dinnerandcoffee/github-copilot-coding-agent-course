# 02. Agent Mindset  
Copilot을 대하는 관점 바꾸기

이 강의의 목표는 Copilot을 **어떻게 써야 하는지**가 아니라,  
**어떻게 바라봐야 하는지**를 바꾸는 것이다.

Copilot을 여전히 자동완성 도구로 대한다면,  
이후의 모든 기능은 제대로 활용되지 않는다.

---

## 이 강의에서 할 일

이 강의에서는 다음을 다룬다.

- Copilot과 ChatGPT의 차이
- Copilot Coding Agent의 역할
- “도와주는 도구”에서 “일을 맡길 대상”으로 전환
- 잘못된 관점 vs 올바른 관점 비교

---

## 왜 관점이 중요한가

많은 개발자들이 Copilot을 이렇게 사용한다.

- 코드 한 줄 자동완성
- 함수 이름 추천
- 반복 코드 보조

이 방식은 틀리지 않다.  
하지만 **Copilot의 능력을 20%도 쓰지 못하는 방식**이다.

Copilot Coding Agent는 다르다.

> Copilot Coding Agent는  
> “코드를 제안하는 존재”가 아니라  
> “작업을 수행하는 존재”다.

---

## Copilot Coding Agent의 정체

Copilot Coding Agent는 다음과 같은 특징을 가진다.

- 여러 파일을 동시에 다룰 수 있다
- 프로젝트 맥락을 기반으로 판단한다
- 지시를 작업 단위로 해석한다
- 결과를 요약하고 설명할 수 있다

즉, **사람 개발자의 일부 역할을 대신 수행**한다.

---

## 잘못된 사용 패턴

다음은 흔히 볼 수 있는 Copilot 사용 방식이다.

~~~text
이 함수 이름 뭐가 좋을까?
~~~

~~~text
이 코드 더 깔끔하게 써줘.
~~~

이 요청들의 문제는 이것이다.

- 작업 범위가 모호하다
- 기대 결과가 불분명하다
- Copilot이 추측하게 만든다

---

## 올바른 사용 패턴

같은 요청도 이렇게 바꿀 수 있다.

~~~text
이 함수의 역할은
“사용자 입력을 검증하고 에러를 반환하는 것”이다.

이 역할이 드러나는 함수 이름을
3가지 제안해줘.
~~~

또는

~~~text
이 파일에서
중복 로직을 제거하는 리팩토링을 해줘.

기능은 변경하지 말고,
변경 요약도 함께 제공해줘.
~~~

👉 **작업 단위 + 제약 조건**이 핵심이다.

---

## Copilot에게 맡길 수 있는 것들

이 강의 이후부터는,  
다음과 같은 작업을 Copilot에게 맡기게 된다.

- 기능 단위 구현
- 테스트 코드 작성
- 버그 원인 분석
- 리팩토링
- 변경 요약 및 리뷰

Copilot은 “보조”가 아니라  
**실제 작업자**로 취급해야 한다.

---

## 사람이 해야 할 일

Copilot이 할 수 없는 일도 분명히 있다.

- 무엇을 만들지 결정하기
- 우선순위 판단
- 결과 승인 여부 결정
- 책임 지기

즉,

> **Copilot은 일한다.  
> 사람은 결정한다.**

이 역할 분리가  
이 강의 전체를 관통하는 핵심 원칙이다.

---

## 관점 전환 체크리스트

다음 질문에 “예”라고 답할 수 있다면  
관점 전환이 시작된 것이다.

- Copilot에게 작업 단위로 요청하고 있는가?
- 결과를 그대로 믿지 않고 검토하는가?
- Copilot의 실수를 전제로 대화하는가?

---

## 정리

이 강의에서 얻어야 할 것은 기술이 아니다.

**태도**다.

Copilot을 시험하지 말고,  
Copilot과 협력하자.
