# 03. Talk to Agent  
말로 개발하기: Copilot과의 대화 워크플로우

이 강의에서는 드디어  
**Copilot Coding Agent와 본격적으로 ‘함께 개발’**하기 시작한다.

키보드로 코드를 치는 비중은 줄이고,  
**말로 작업을 지시하는 비중**을 늘린다.

---

## 이 강의의 목표

이 강의가 끝나면 다음을 할 수 있어야 한다.

- Copilot에게 작업을 단계적으로 지시한다
- 한 번에 완벽한 결과를 기대하지 않는다
- 대화를 통해 결과를 점점 개선한다
- “지시 → 결과 → 수정” 루프를 자연스럽게 만든다

---

## 왜 ‘말로 개발’해야 하는가

전통적인 개발 흐름은 이렇다.

1. 설계
2. 코드 작성
3. 실행
4. 수정

Copilot Coding Agent를 쓰면 흐름이 바뀐다.

1. 작업 지시
2. 결과 확인
3. 추가 지시
4. 정제

즉,

> **개발의 중심이  
> 코드 작성에서 대화로 이동한다.**

---

## Step 1. 아주 작은 작업부터 지시하기

처음부터 큰 기능을 맡기지 않는다.  
아주 작은 작업으로 시작한다.

~~~text
이 프로젝트에
간단한 TODO 기능을 추가하려고 해.

먼저 TODO 항목 하나를
추가할 수 있는 함수 초안만 만들어줘.
~~~

### 관찰 포인트

- Copilot이 작업 범위를 이해하는가?
- “초안”이라는 표현을 인식하는가?

결과가 마음에 들지 않아도 괜찮다.  
**대화는 이제 시작이다.**

---

## Step 2. 결과를 전제로 다시 지시하기

Copilot의 첫 결과를 그대로 두지 않는다.  
그 결과를 **전제**로 삼아 다시 말한다.

~~~text
좋아, 이 방향은 맞아.

다만 다음을 수정해줘.
- 함수 이름을 더 명확하게
- 입력 검증 로직 추가
- 변경 요약 제공
~~~

👉 중요한 점은  
**“다시 처음부터”가 아니라 “이 결과를 기준으로”**다.

---

## Step 3. 범위를 명확히 제한하기

Copilot이 과하게 움직일 때가 있다.  
이때는 범위를 명확히 잘라준다.

~~~text
이 작업은
이 파일 안에서만 진행해줘.
다른 파일은 수정하지 말아줘.
~~~

이 한 문장으로  
**위험도가 급격히 낮아진다.**

---

## Step 4. 일부러 모호한 지시도 해보기

이번에는 일부러 애매하게 요청해보자.

~~~text
이 코드를 더 좋게 만들어줘.
~~~

그리고 스스로에게 질문해본다.

- Copilot은 무엇을 추측했는가?
- 어디까지 수정하려 했는가?

이 경험을 통해  
**왜 좋은 지시가 필요한지 몸으로 이해**하게 된다.

---

## 좋은 대화의 기준

Copilot과의 대화가 잘 되고 있다는 신호는 다음과 같다.

- Copilot이 질문을 되묻는다
- 변경 요약을 먼저 제안한다
- 한 번에 모든 걸 바꾸지 않는다

반대로, 위험 신호는 이것이다.

- 이유 없는 대규모 수정
- 설명 없는 결과 제시
- 과도한 추측

---

## 대화형 개발 패턴 요약

이 강의에서 배운 기본 패턴은 이것이다.

1. 작은 작업 지시
2. 결과 확인
3. 구체적인 수정 요청
4. 범위 제한
5. 반복

이 패턴은  
이후 모든 강의에서 계속 사용된다.

---

## 정리

말로 개발한다는 것은  
코딩을 안 한다는 뜻이 아니다.

> **코딩의 ‘순서’를 바꾼다**는 뜻이다.

먼저 말하고,  
그 다음 코드를 본다.
