# 09. Build · Test · Ship  
기능 개발을 Copilot에게 맡기는 법

이 강의에서는 Copilot Coding Agent에게  
**기능 하나를 처음부터 끝까지 맡기는 경험**을 한다.

이 단계에 오면 Copilot은 더 이상 보조가 아니다.  
**실제 개발 워커**다.

---

## 이 강의의 목표

이 강의가 끝나면 다음을 할 수 있어야 한다.

- 기능 단위로 작업을 위임한다
- 완료 기준(Definition of Done)을 제시한다
- Build · Test · Ship 흐름을 Copilot과 함께 운영한다
- 사람이 개입해야 할 지점을 명확히 안다

---

## 기능 위임의 기본 원칙

기능을 맡길 때 가장 중요한 것은 이것이다.

> **“무엇을 만들 것인가”보다  
> “언제 끝난 것으로 볼 것인가”를 명확히 하는 것**

Copilot은 기준이 명확할수록  
훨씬 안정적으로 일한다.

---

## Step 1. 기능을 명확히 정의하라

먼저 기능 요구사항을 정리한다.

~~~text
다음 기능을 구현해줘.

기능:
- TODO 항목을 삭제하는 기능

요구사항:
- ID로 TODO를 삭제
- 존재하지 않는 ID는 에러 반환

완료 기준:
- 정상 삭제 테스트 통과
- 에러 케이스 테스트 통과
- 변경 요약 제공
~~~

👉 이 단계에서  
이미 절반은 성공이다.

---

## Step 2. 바로 코드를 짜게 하지 말자

위험한 기능일수록  
계획부터 받는다.

~~~text
바로 구현하지 말고,
이 기능을 구현하기 위한
작업 단계와 수정 대상 파일을
먼저 설명해줘.
~~~

이 답변을 보고  
사람이 한 번 판단한다.

---

## Step 3. 구현 + 테스트를 함께 맡기기

이제 실제 작업을 맡긴다.

~~~text
앞서 설명한 계획에 따라
기능 구현과 테스트 작성을 진행해줘.

조건:
- 테스트 먼저 작성
- 모든 테스트 통과
- 기존 기능 영향 없음
~~~

Copilot은  
이 패턴에 매우 잘 반응한다.

---

## Step 4. Ship 전에 마지막 검증

구현이 끝나면  
바로 배포하지 않는다.

~~~text
이 변경을 배포해도 되는지
리스크 관점에서 검토해줘.

- 잠재적 문제
- 롤백 시나리오
- 주의사항
~~~

👉 이 단계가  
사람의 판단을 돕는다.

---

## 사람이 개입해야 하는 순간

모든 것을 Copilot에게 맡기지는 않는다.

사람이 반드시 개입해야 하는 순간은:

- 요구사항이 모호할 때
- 비즈니스 로직 판단이 필요할 때
- 보안/성능 영향이 있을 때

> **Copilot은 실행자,  
> 사람은 책임자다.**

---

## 흔한 실수

- 완료 기준 없이 기능 요청
- “적당히 잘 만들어줘” 요청
- 테스트를 선택 사항으로 취급

이 경우 Copilot은  
불안정한 결과를 낼 확률이 높다.

---

## 기능 위임 체크리스트

- [ ] 기능 요구사항이 명확하다
- [ ] 완료 기준이 정의되어 있다
- [ ] 계획 → 구현 → 테스트 흐름을 거쳤다
- [ ] 배포 전 검증을 했다

이 체크리스트를 통과하면  
기능을 맡겨도 된다.

---

## 정리

Copilot에게 기능을 맡긴다는 것은  
편해지기 위한 선택이 아니다.

> **더 정확하게 일하기 위한 선택**이다.
